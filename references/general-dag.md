# 通用 DAG 模板

当用户目标不匹配 `code_dev` 或 `deep_research` 场景时，Coordinator 使用通用 DAG 模板动态推断任务结构。

## 触发条件

不匹配以下场景时自动使用：
- `code_dev`: 实现/开发/重构/写代码/修bug/添加功能
- `deep_research`: 研究/调查/分析/报告/对比/总结

## 动态推断原则

Coordinator 根据以下原则设计 DAG：

1. **无依赖 → 并行**: 能同时跑的绝不串行
2. **有依赖 → blockedBy**: 上游完成才触发下游
3. **最大并行度 10**: 避免 API 速率限制
4. **单一职责**: 每个子任务只做一件事，独立可验证
5. **适度粒度**: 2-10 个子任务，避免过度碎片化

## 通用 DAG 结构

```
Coordinator 分析目标
       │
       ▼
  识别子任务间的依赖关系
       │
       ├── 独立子任务 → 并行组
       └── 依赖子任务 → 串行链
              │
              ▼
         生成动态 DAG
```

## 常见通用场景示例

### 示例 1: 数据迁移

```
用户: "把 MySQL 的用户数据迁移到 PostgreSQL"

Coordinator 拆解:
  T1: 分析 MySQL schema（无依赖）
  T2: 设计 PostgreSQL schema（依赖 T1）
  T3: 编写迁移脚本（依赖 T2）
  T4: 数据校验脚本（依赖 T2）
  T5: 执行迁移测试（依赖 T3, T4）
  T6: 生成迁移报告（依赖 T5）

DAG: T1 → T2 → [T3, T4 并行] → T5 → T6
```

### 示例 2: 配置管理

```
用户: "帮我给这个项目添加 CI/CD 配置和 Docker 支持"

Coordinator 拆解:
  T1: 分析项目结构和依赖（无依赖）
  T2: 编写 Dockerfile（依赖 T1）
  T3: 编写 docker-compose.yml（依赖 T2）
  T4: 编写 CI 配置文件（依赖 T1）
  T5: 编写 CD 部署脚本（依赖 T3）
  T6: 验证完整流程（依赖 T2, T3, T4, T5）

DAG: T1 → [T2, T4 并行] → T3 → T5 → T6
     注意: T4 不依赖 T2/T3，可以和 T2 并行
```

### 示例 3: 文档生成

```
用户: "为这个项目的所有 API 生成文档"

Coordinator 拆解:
  T1: 扫描所有 API 端点（无依赖）
  T2: 分析 REST API 文档（无依赖）
  T3: 分析 GraphQL 文档（无依赖）
  T4: 生成 API 参考文档（依赖 T1, T2, T3）
  T5: 生成使用示例（依赖 T1）
  T6: 汇总文档站点（依赖 T4, T5）

DAG: [T1, T2, T3 并行] → [T4, T5 并行] → T6
```

## 通用 Agent Prompt 模板

```
你的任务是：<任务描述>

上下文：<Coordinator 提供的相关上下文>
输入：<依赖的上游输出文件路径>
输出：<期望的输出格式和文件路径>

要求：
1. 只做本任务范围内的事
2. 完成后将结果写入指定路径
3. 遇到无法处理的依赖缺失时，标注清楚
```

## 自适应调整

通用 DAG 任务完成后，Coordinator 执行 Replan Check：

```
已完成任务输出
  │
  ▼
评估:
  ├── 输出正常 → 继续
  ├── 输出异常复杂 → Split 下游任务
  ├── 两个上游高度重叠 → Merge 下游任务
  ├── 发现遗漏 → Append 新任务
  └── 前提错误 → Replan（HITL 确认）
```

## 与特定场景的区别

| 维度 | 特定场景（code_dev/deep_research） | 通用 DAG |
|------|----------------------------------|---------|
| 模板化程度 | 高（固定阶段和验证门禁） | 低（动态设计） |
| 验证强度 | 按 SOP 预定义 | Coordinator 按风险判断 |
| HITL 密度 | SOP 预设审批点 | 动态识别需要审批的节点 |
| 适用任务数 | 3-10 | 2-8（通常更简单） |
